## Conventions

- **Project scope.** These skills live in `.claude/skills/` and are committed, so they apply to every contributor and every subagent — not just your machine.
- **Directory name == command == `name`.** The kebab-case folder is the `/command` and must equal frontmatter `name`. Bundled `references/`, `scripts/`, `examples/`, and `assets/` sit exactly one level deep and load on demand.
- **Validate before committing.** Run `python3 ~/.claude/plugins/marketplaces/claude-plugins-official/plugins/skill-creator/skills/skill-creator/scripts/quick_validate.py <skill-dir>`. A warning for the Claude Code extension fields (`disable-model-invocation`, `argument-hint`) is expected and fine.
- **Two special kinds.** `run-codegen` and `run-migration` carry `disable-model-invocation: true` — side-effecting workflows Claude never fires on its own; you trigger them by name. `audit-offline-integrity` is a **subagent** in `.claude/agents/`, invoked through the Task tool, not a SKILL.md loaded into the main context.
- **Shared vocabulary.** Every skill uses the same words on purpose: *feature folder*, *repository*, *Notifier/ViewModel*, *design token*, *value object*, *Result/Failure*, *canonical units*. Keep new skills consistent with it.

## Relationship to CLAUDE.md and docs/

- **`docs/` is the source of truth.** These skills are the operational distillation of the design and feature dossiers under `docs/` (`docs/flutter/`, `docs/design/pulse/`, `docs/features/`, `docs/reference/`). Many skills cite their originating doc in frontmatter `metadata.source-docs` — read the doc for the *why*, invoke the skill for the *how* and the enforceable rules.
- **CLAUDE.md is the always-on baseline; skills are the deep, on-demand layer.** CLAUDE.md carries the small set of rules that must be in context for every turn; a skill loads its full playbook only when your task matches its triggers, keeping the base context lean. When a rule appears in both, they must agree — update CLAUDE.md and the relevant skill together.
