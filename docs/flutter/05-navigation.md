# 🧭 Navigation & Routing

> How screens are declared, deep-linked, guarded, and restored in Car and Pain — with local-notification taps treated as the app's real deep links.

📍 Part of the **[Flutter Engineering Guide](./README.md)** · See also **[Local Notifications & Background Reliability](./07-notifications.md)** · **[Architecture & Module Structure](./01-architecture-and-structure.md)** · **[Internationalization, RTL, Calendars & Numerals](./06-i18n-rtl-calendars.md)**

## Decision

We use **go_router** (first-party, pin the verified current-stable major at kickoff — the research baseline is v17.3.x) as the app's single routing layer, with **type-safe routes via `go_router_builder`** (`@TypedGoRoute` + `GoRouteData`). The shell is one **`StatefulShellRoute.indexedStack`** whose branches are the ~6 top-level modules, each an internal master-detail stack. Full-screen flows are pushed above the shell via a `rootNavigatorKey`, guards are **declarative `redirect`** functions, and `restorationScopeId` is wired end-to-end. Notification payloads map to screens through a **pure `payload → location` function**, with `getNotificationAppLaunchDetails()` feeding `initialLocation` for cold starts.

## Why

go_router is the Flutter-team-recommended router, and its **first-party long-term maintenance** is the deciding factor for a buy-once app that must survive years without a server to hotfix it. It is currently in a stability/bug-fix phase (feature-complete, not abandoned) — exactly what we want.

Alternatives considered and rejected:

- **auto_route** — *fallback only.* Its typed nested/tab routes are more ergonomic and `AutoTabsRouter`/`AutoRouteGuard` are purpose-built, but it is single-maintainer (weaker longevity for a decade-long product), forces `build_runner` codegen for *all* routes, and its standout strength — rich imperative guards and deep nesting — is wasted here because there is **no auth** and the nesting is shallow (tabs + master-detail). Keep it in the back pocket only if go_router's typed nested routes become a real blocker.
- **Raw Navigator 2.0 / Router API** — *reject.* You would reimplement deep-link parsing, redirects, shell persistence, and restoration that go_router already ships; officially discouraged; a maintenance liability for a solo/small team.
- **Named routes / `onGenerateRoute`** — *reject for the app shell.* No first-class deep linking (we need notification taps), no declarative stack reconstruction, no shell/tab persistence, no type safety; does not scale to ~25 modules. Fine only for ephemeral pageless UI (dialogs, sheets), which coexists with go_router.
- **Beamer** — *reject.* Smaller momentum, not first-party; go_router covers the same ground with official backing.

## How we do it

### Folder layout

The router is a single, small, stable module in the app shell:

```text
apps/car_and_pain/lib/src/routing/
  app_router.dart          # the one GoRouter + rootNavigatorKey + branch keys
  routes.dart              # @TypedGoRoute / GoRouteData declarations
  routes.g.dart            # generated by go_router_builder (git-ignored, CI-regenerated)
  notification_router.dart # PURE payload -> location mapper (Flutter-free where possible)
  guards.dart              # pure redirect helpers (onboarding, has-vehicle)
```

### The single GoRouter

```dart
final _rootNavigatorKey = GlobalKey<NavigatorState>(debugLabel: 'root');
final _dashboardKey     = GlobalKey<NavigatorState>(debugLabel: 'dashboard');
final _vehiclesKey      = GlobalKey<NavigatorState>(debugLabel: 'vehicles');
// ...one navigatorKey per branch

GoRouter buildAppRouter({
  required Listenable refresh,      // fed by Riverpod onboarding/vehicle-count state
  required String initialLocation,  // from getNotificationAppLaunchDetails(), else '/'
}) {
  return GoRouter(
    navigatorKey: _rootNavigatorKey,
    initialLocation: initialLocation,
    restorationScopeId: 'app_router',
    refreshListenable: refresh,
    redirect: appRedirect,          // pure function, see guards.dart
    routes: $appRoutes,             // generated aggregate from routes.g.dart
  );
}
```

`MaterialApp.router` wires it up with restoration enabled:

```dart
MaterialApp.router(
  restorationScopeId: 'app',
  routerConfig: appRouter,
  locale: appLocale,               // app-controlled locale drives Directionality/RTL
  localizationsDelegates: AppLocalizations.localizationsDelegates,
  supportedLocales: AppLocalizations.supportedLocales,
);
```

### Tab shell — `StatefulShellRoute.indexedStack`

Each of the ~6 top-level modules is a branch; `indexedStack` keeps every tab's navigation stack **and scroll position alive** across switches.

```dart
StatefulShellRoute.indexedStack(
  builder: (context, state, navigationShell) =>
      ScaffoldWithNavBar(navigationShell: navigationShell),
  branches: [
    StatefulShellBranch(navigatorKey: _dashboardKey, routes: [ /* Dashboard */ ]),
    StatefulShellBranch(navigatorKey: _vehiclesKey,  routes: [ /* Vehicles master-detail */ ]),
    // Reminders, Costs/TCO, Reports/Charts, Settings ...
  ],
);
```

Switch tabs without rebuilding the branch:

```dart
navigationShell.goBranch(
  index,
  initialLocation: index == navigationShell.currentIndex, // re-tap resets to branch root
);
```

### Master-detail with stable path-param IDs

Detail screens carry **stable DB IDs in the path**, never in `extra`, so any screen is reconstructable from its URL alone:

```text
/vehicles                                   (list)
/vehicles/:vehicleId                        (detail)
/vehicles/:vehicleId/reminders/:reminderId  (deep-linkable target)
```

### Full-screen flows above the shell

Add/edit vehicle, add reminder, backup/restore, and the onboarding wizard cover the bottom nav via `parentNavigatorKey`:

```dart
GoRoute(
  path: 'new',
  parentNavigatorKey: _rootNavigatorKey, // renders ABOVE the tab bar
  builder: (context, state) => const AddVehicleScreen(),
);
```

### Type-safe routes (from day one)

```dart
@TypedGoRoute<VehicleDetailRoute>(
  path: '/vehicles/:vehicleId',
  routes: [TypedGoRoute<ReminderDetailRoute>(path: 'reminders/:reminderId')],
)
class VehicleDetailRoute extends GoRouteData with _$VehicleDetailRoute {
  const VehicleDetailRoute({required this.vehicleId});
  final String vehicleId;

  @override
  Widget build(BuildContext context, GoRouterState state) =>
      VehicleDetailScreen(vehicleId: vehicleId);
}

// Navigate with compile-time safety:
const VehicleDetailRoute(vehicleId: id).go(context);
```

Prefer `.go()` over `.push()` (see Pitfalls — `push()` stopped reflecting in the URL since go_router 8.0).

### Declarative guards

The guard surface is tiny — onboarding-complete and has ≥1 vehicle — so a pure `redirect` fully covers it:

```dart
String? appRedirect(BuildContext context, GoRouterState state) {
  final loc = state.matchedLocation;
  if (!onboardingComplete && loc != '/onboarding') return '/onboarding';
  if (noVehiclesYet && needsVehicle(loc) && loc != '/vehicles/new') {
    return '/vehicles/new';
  }
  return null; // no redirect
}
```

Make guards reactive by feeding a Riverpod-backed `Listenable` to `refreshListenable` so they re-run when onboarding/vehicle-count state changes.

### Notification deep-link bridge

This is the highest-value, fully unit-testable seam. A **pure** mapper turns a payload into a location; cold start and warm taps use the same mapper.

```dart
// notification_router.dart — pure, no BuildContext, no plugins
String mapNotificationPayload(String? payload) {
  final p = NotificationPayload.tryParse(payload);
  return switch (p) {
    ReminderPayload(:final vehicleId, :final reminderId) =>
      '/vehicles/$vehicleId/reminders/$reminderId',
    _ => '/', // unknown/legacy payload falls back to dashboard
  };
}
```

Wire cold start into `initialLocation` and warm taps into `router.go()`:

```dart
// In bootstrap(), BEFORE runApp:
final launch = await flnp.getNotificationAppLaunchDetails();
final initial = (launch?.didNotificationLaunchApp ?? false)
    ? mapNotificationPayload(launch!.notificationResponse?.payload)
    : '/';

// In the FLN init callback (background/foreground taps):
onDidReceiveNotificationResponse: (resp) =>
    appRouter.go(mapNotificationPayload(resp.payload));
```

Because the target route only needs the IDs, the screen rebuilds its data from the encrypted SQLite DB — surviving reboot/Doze/cold-start where `extra` would be null. See **[Local Notifications & Background Reliability](./07-notifications.md)**.

### Packages

```yaml
dependencies:

```

## Rules

- **DO** carry deep-linkable identity in **path params** (`:vehicleId`, `:reminderId`); **DON'T** rely on `state.extra` for anything a notification, deep link, or restore can reach — it is null on cold start.
- **DO** prefer `.go()` / typed `.go()`; **DON'T** use `push()` for shell navigation (URL/redirect location stops reflecting it since go_router 8.0). Only set `optionURLReflectsImperativeAPIs = true` if you truly need push semantics reflected.
- **DO** keep `redirect` a **pure, idempotent** function and always exclude the target location (`loc != '/onboarding'`); **DON'T** write a redirect that can always return a non-matching location (infinite loop).
- **DO** set `parentNavigatorKey: _rootNavigatorKey` on every full-screen flow; **DON'T** let add/edit forms nest inside a branch (they render under the bottom nav).
- **DO** assign explicit `navigatorKey`s per branch; **DON'T** share or omit them (breaks per-tab back behavior).
- **DO** use go_router for anything screen-like and deep-linkable; **DON'T** route ephemeral UI (confirm dialogs, pickers, snackbars) through it — use plain `showDialog`/`showModalBottomSheet`.
- **DO** guard `context` across async gaps: `if (!context.mounted) return;` before `context.go`, or capture `GoRouter.of(context)` before the `await`.
- **DO** use `PopScope` (`canPop` + `onPopInvokedWithResult`) / go_router `onExit` for confirm-discard on unsaved forms; **DON'T** use `WillPopScope` on page-backed screens.
- **DON'T** add `FlutterDeepLinkingEnabled`/App-Links config — routing is driven in Dart from notification payloads, not OS deep links; adding it invites handler conflicts.
- **CI** runs `build_runner build` then `build_verify` so typed-route codegen never drifts; generated `*.g.dart` is git-ignored and regenerated as the first CI step (see **[Build, Tooling, Release & CI/CD](./12-build-ci-release.md)**).

## For Car and Pain specifically

- **Offline / account-free** collapses the guard surface to two redirects (onboarding-complete, has-vehicle), which is precisely why declarative `redirect` is enough and auto_route's guard machinery is unnecessary.
- **Notifications are the deep links.** The reminder engine (time + odometer-projection distance + engine-hour triggers) fires taps that must land on `/vehicles/:vehicleId/reminders/:reminderId` reconstructed from the encrypted DB alone — after a reboot or OEM battery-kill, `extra` is gone, so path-param identity is non-negotiable. The router shares `rootNavigatorKey` so notification handling works with no live `BuildContext`.
- **~25 feature modules map to ~6 shell branches** (Dashboard, Vehicles, Reminders, Costs/TCO, Reports/Charts, Settings); heavy flows (import/export, TCO wizard, chart drill-downs) sit as full-screen routes above the shell.
- **RTL is not the router's job** — locale/Directionality live at `MaterialApp` level (see **[Internationalization, RTL, Calendars & Numerals](./06-i18n-rtl-calendars.md)**). Material/Cupertino page transitions already mirror automatically; only *custom* transitions must read `Directionality.of(context)`.
- **State restoration serves reboot/Doze survival** for in-progress edit forms and the last-viewed screen — the same durability posture as the rest of the app.
- **No-telemetry** is unaffected by routing; we drive everything in Dart and add zero OS deep-link/analytics surface.

## Testing

Aligns with the guide's logic-heavy pyramid — the router's pure seams are unit-tested exhaustively.

- **Pure mapper (highest value):** table-driven unit tests on `mapNotificationPayload` — valid `ReminderPayload`, malformed/legacy payloads, null → `/` — this is the reboot-safe-reminder guarantee.
- **Pure redirect:** given `(app-state, location) → expected redirect`, assert no loops and correct onboarding/has-vehicle bounces.
- **Router in isolation:** build the `GoRouter`, call `router.go(uri)` / typed `Route(...).go()`, assert `router.routerDelegate.currentConfiguration` / `router.state.uri`.
- **Cold-start / deep link:** construct the router with `initialLocation` set to a mapped location, seed an **in-memory Drift DB**, and assert the target screen renders with data rebuilt via the `extra == null` path.
- **Shell behavior (widget test):** pump `MaterialApp.router`, tap across `StatefulShellRoute` tabs, and verify each branch keeps its own stack and scroll.
- **RTL:** wrap in `Directionality(textDirection: TextDirection.rtl)` (or force a fa/ar/ckb locale); golden the shell + a representative transition to confirm mirroring — exercise both platforms via `debugDefaultTargetPlatformOverride`.
- **State restoration:** `tester.restartAndRestore()` / `tester.getRestorationData()` to confirm the last route and `RestorationMixin` form fields return after an OS kill.
- **CI:** run `build_runner` + `build_verify` so typed-route codegen never drifts.

See **[Testing Strategy](./11-testing.md)** for the shared harness (deterministic `Clock`, RTL/locale goldens, `ProviderContainer` overrides).

## Pitfalls

- **`extra` is not deep-linkable or restorable.** After a notification cold-start, reboot, or OS restore, `state.extra` is null. Every notification/deep-link-reachable screen must accept an ID and rebuild from the DB. Pass rich objects as `extra` only for in-session convenience, and always fall back to rebuilding from the ID.
- **`push()` and the URL.** Since go_router 8.0, imperative `push()` no longer updates the location that redirects/URL observe. Prefer `.go()`; avoid `push()` for shell navigation.
- **Redirect infinite loops.** Forgetting the `state.matchedLocation != target` guard loops forever. Keep `redirect` pure and idempotent.
- **`indexedStack` keeps all branches alive.** Fine for ~6 tabs, but don't stuff dozens of heavy branches in one shell; lazy-build detail pages and watch chart widgets holding large datasets in inactive tabs (see **[Performance & Rendering](./10-performance-rendering.md)**).
- **Nested navigator-key mistakes.** Missing `parentNavigatorKey` renders full-screen forms under the bottom nav; a wrong branch `navigatorKey` breaks per-tab back. Assign keys explicitly.
- **Codegen drift.** Editing a route without regenerating `*.g.dart` produces confusing compile errors — CI `build_verify` catches it; keep routes in a small, stable module.
- **RTL in custom transitions.** A `SlideTransition` hard-coding `Offset(1, 0)` slides the wrong way in fa/ar/ckb. Read `Directionality.of(context)` or lean on the built-in directionality-aware transitions.
- **Deep-linking flag conflicts.** If a native/plugin deep-link handler is ever added, disable Flutter's built-in handler or the two fight. For local-notification-only routing, don't add App-Links/Universal-Links config at all.
- **`BuildContext` across async gaps.** After awaiting a DB/backup op, guard with `context.mounted` or capture `GoRouter.of` before the `await`.

## Decisions to confirm

- **State management binding:** the router's `refreshListenable` and guard state assume Riverpod. If the solo developer's existing **flutter_bloc** muscle memory wins at kickoff, the `refreshListenable` source (a `Cubit`/`Bloc`-backed `Listenable`) changes accordingly — confirm before locking route wiring. See **[State Management with Riverpod](./02-state-management.md)**.
- **Household peer-to-peer sync:** if this moves *into* MVP scope, multi-device targets may need additional route identity/merge awareness. Confirm it stays out of MVP so route IDs can remain single-device UUIDv7 path params.
